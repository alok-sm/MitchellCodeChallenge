Mitchell Coding challenge

This is the repository for the code written for the Mitchell coding challenge. The Challenge Document required us to build a CRUD API for a vehicle class with automated testing and in memory storage of objects.

Architecture

The solution was architected with four ideas in mind:

  • Separation of CRUD interface and the "Vehicle" type

    This is because most production applications use the CRUD interface several times and generalizing the CRUD aspect of the API would be useful if we were to write an endpoint for another class. This is how you would create a new ComputerController class:

    public class ComputerController: CrudControllerBase public class ComputerModel : CrudModelBase { public string OemName { get; set; } public string Model { get; set; } }

  • Separation of data storage and application

    The challenge asks us to implement an in memory storage but in production application, we would use some sort of a database to store data. Hence, separating the interface and the implementation for the data storage is important

  • Easy extensibility of the CRUD interfact

    The CRUD interface can easily be extended to incorporate validation, etc. The controller class can override a virtual function on the CrudControllerBase class, do the validations and then (optionally) give the base class control to do the remaining job

  • Automated Unit Testing and API endpoint testing

The API is hosted on http://mitchellchallenge.azurewebsites.net.
